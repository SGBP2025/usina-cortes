# Story EPIC-001.4: Setup Middleware (Auth, Rate Limit, Segurança)

## Metadata
- **Story ID:** EPIC-001.4
- **Epic:** EPIC-001 — Transformação SaaS
- **Sprint:** 1 — Foundation
- **Points:** 8
- **Status:** Approved
- **Priority:** P0 — Garante que nenhuma rota fica desprotegida
- **Depende de:** EPIC-001.1, EPIC-001.2, EPIC-001.3

## Story
**As a** sistema  
**I want** um middleware que proteja rotas autenticadas, aplique rate limiting e headers de segurança  
**So that** nenhuma API ou página do dashboard seja acessível sem autenticação válida

## Acceptance Criteria
- [ ] AC1: `middleware.ts` protege todos os paths `/dashboard/*` e `/api/*` (exceto `/api/auth/*`)
- [ ] AC2: Usuário sem sessão válida é redirecionado para `/auth/login` com `?redirectTo=` original
- [ ] AC3: Rate limiting em `/api/jobs/create`: máximo 5 jobs/hora por usuário
- [ ] AC4: Headers de segurança configurados: `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Content-Security-Policy`
- [ ] AC5: `/api/*` retorna JSON com status 401 (não redirect) quando sem auth
- [ ] AC6: Tokens de sessão expirados são renovados automaticamente via middleware
- [ ] AC7: Logs de acesso negado registrados (sem expor dados sensíveis)

## Dev Notes

**Decisão Aria:** Rate limiting usando Redis (Upstash). O `REDIS_URL` já está no `.env.example` da Story 1.1. Para desenvolvimento local, pode usar `REDIS_URL=redis://localhost:6379`.

**CRÍTICO Quinn (ponto 5 do checklist):** Rate limiting é obrigatório em `/api/jobs/create`. Créditos insuficientes devem ser verificados ANTES de enfileirar o job (validado na Story 2.2, mas o middleware não impede — é uma validação de negócio).

**CSP inicial:** Começar com policy permissiva e apertar gradualmente. Supabase Storage URLs precisam estar no CSP para `img-src`. Stripe.js precisa estar no `script-src`.

**Exceções no matcher:** `/api/auth/callback` (Supabase OAuth callback) e `/api/webhooks/stripe` devem estar FORA do matcher de auth. Stripe verifica assinatura HMAC própria.

**Renovação de sessão:** O `lib/supabase/middleware.ts` do Supabase SSR cuida disso automaticamente via `supabase.auth.getSession()` — simplesmente chamá-lo no middleware é suficiente.

## Testing

- Teste de auth: acessar qualquer rota protegida sem cookie → redirect correto
- Teste de rate limit: script que faz 6 POSTs seguidos → confirmar 429 no 6º
- Teste de headers: `curl -I https://localhost:3000` → confirmar X-Frame-Options presente
