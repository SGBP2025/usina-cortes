## Dev Notes

**Decisão Aria:** Usar **Supabase Auth nativo** — NÃO implementar JWT customizado como sugerido no DB-AUDIT.md original. O Supabase Auth já provê JWT, refresh tokens e integração direta com RLS via `auth.uid()`.

**CRÍTICO Quinn:** RLS deve estar ativa ANTES de qualquer dado de usuário ser inserido. A migration deve incluir as policies RLS no mesmo arquivo, não em migration separada.

**Status ENUM:** Usar `CREATE TYPE job_status AS ENUM ('pending', 'processing', 'completed', 'error', 'cancelled')`. Se Lovable/cliente precisar de valores adicionais, adicionar via migration, nunca modificar o tipo diretamente.

**Worker access:** O worker (rodando no Railway) usará `SUPABASE_SERVICE_ROLE_KEY` que bypassa RLS. Isso é intencional — o worker acessa dados de qualquer usuário para processar os jobs. Documentar isso explicitamente nos comentários do código.

**Supabase Storage:** Criar bucket `videos` (private) e `clips` (private) no Supabase Storage durante esta story. Configurar políticas de storage para acesso apenas do próprio usuário.
